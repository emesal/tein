# tein development handoff

> *branch and rune-stick* — embeddable chibi-scheme for rust

## project status

### ✅ completed
- **core infrastructure**
  - vendored chibi-scheme 0.11 with custom build system
  - c ffi shim layer (`tein_shim.c`) for macro-based apis
  - safe rust wrappers around unsafe c functions
  - proper memory management (drop trait for contexts)

- **value types implemented**
  - integers (`i64`)
  - floats (`f64`)
  - strings (`String`)
  - symbols (`String`)
  - booleans (`bool`)
  - lists (`Vec<Value>`) - proper lists ending in nil
  - pairs (`Box<Value>, Box<Value>`) - improper lists / dotted pairs
  - vectors (`Vec<Value>`) - scheme `#(...)` vectors
  - nil (empty list)

- **api**
  - `Context::new()` - creates scheme evaluation context
  - `Context::evaluate(code: &str)` - evaluates scheme expressions
  - `Context::define_fn0..3(name, f)` - register rust functions in scheme
  - `Value` enum with safe extraction
  - `Value::to_raw()` - convert rust values back to scheme
  - `raw` module - re-exports ffi helpers for foreign functions
  - proper error handling via `Result<T, Error>`
  - detailed error messages (extracts exception message + irritants)

- **testing**
  - 20 unit tests covering all types, vectors, errors, and ffi
  - 3 doc-tests (lib.rs, Context, Context::evaluate)
  - working examples (`basic.rs`, `floats.rs`, `debug.rs`, `ffi.rs`)
  - all tests passing ✓

- **code review fixes (2026-02-05)**
  - fixed critical type safety issue (sexp_uint_t signedness)
  - removed unsound Send impl, documented !Send + !Sync rationale
  - added tortoise-and-hare cycle detection for circular lists
  - depth limit (MAX_DEPTH = 10_000) prevents stack overflow
  - derived PartialEq on Value for ergonomic testing
  - to_raw returns Result, errors clearly on unsupported types
  - fixed broken rustdoc links
  - string Display properly escapes special chars
  - transmute has explicit type annotations
  - comprehensive Safety docs in ffi module

### 🚧 known limitations

1. **no r7rs standard environment**
   - currently running with minimal chibi primitives only
   - basic arithmetic, cons/car/cdr, quote, make-vector work
   - missing: most r7rs standard library functions
   - **why**: requires either:
     - setting up static library embedding (complex)
     - enabling dynamic module loading (conflicts with vendoring goals)

2. **limited type coverage**
   - no hash tables
   - no ports/io
   - no procedures (scheme functions as values)
   - no continuations representation
   - no bytevectors

3. **ffi is low-level**
   - foreign functions use raw `sexp` types
   - no automatic argument/return conversion yet
   - need proc macro for ergonomic api

## architecture

### directory structure
```
tein/
├── src/
│   ├── lib.rs          # public api, re-exports
│   ├── context.rs      # Context type, evaluation
│   ├── value.rs        # Value enum, scheme→rust conversion
│   ├── error.rs        # Error, Result types
│   └── ffi.rs          # unsafe c bindings + safe wrappers
├── vendor/
│   └── chibi-scheme/   # chibi 0.11 source
│       ├── tein_shim.c # our c bridge for macros
│       └── include/chibi/install.h  # generated by build.rs
├── build.rs            # compiles chibi + shim
├── examples/           # usage examples
└── Cargo.toml
```

### key design decisions

**1. vendoring chibi**
- chibi source is bundled in `vendor/`
- compiled during `cargo build` via `build.rs`
- zero external dependencies for users
- `install.h` generated with minimal config

**2. shim layer**
- chibi uses lots of c macros (`sexp_car`, `sexp_integerp`, etc.)
- rust ffi can't call macros directly
- `tein_shim.c` exports them as real functions
- rust calls `tein_sexp_car()` which calls `sexp_car()` macro

**3. type checking order matters!**
- **critical**: check `sexp_flonump` BEFORE `sexp_integerp`
- `sexp_integerp` includes `_or_integer_flonump` so matches 4.0
- wrong order causes floats to be read as garbage integers

**4. unsafe boundaries**
- all ffi in `ffi.rs` is `unsafe`
- `Context` and `Value` provide safe api
- users never touch raw `sexp` pointers

## gotchas & debugging tips

1. **rust 2024 edition strictness**
   - all `unsafe fn` bodies need inner `unsafe { }` blocks
   - caught us multiple times!

2. **chibi context initialization**
   - `sexp_make_eval_context` signature: `(parent, stack, env, size, max_size)`
   - first three are `NULL` for new contexts, NOT sizes!
   - we initially got this wrong → segfault

3. **linker errors for chibi functions**
   - means you forgot to add it to `tein_shim.c`
   - or forgot to wrap it in `ffi.rs`

4. **build.rs changes**
   - cargo doesn't always detect when shim changes
   - `cargo clean` if things get weird

5. **testing scheme code**
   - no repl yet (could be a fun addition!)
   - write small examples/ files for now
   - scheme primitives available: +, -, *, /, cons, car, cdr, quote, basics

## questions for future fey

**if you want to add a new scheme type:**
1. check if chibi has a predicate (e.g., `sexp_vectorp`)
2. add wrapper to `tein_shim.c`
3. add extern declaration to `ffi.rs`
4. add rust wrapper function in `ffi.rs`
5. add variant to `Value` enum
6. add extraction logic in `Value::from_raw`
7. add display impl
8. write test/example

**if you want r7rs:**
- read chibi's `eval.c` around `sexp_load_standard_env`
- understand `sexp_static_libraries` mechanism
- might need to vendor the `.scm` files and generate c arrays?
- alternatively: enable module loading, accept dynamic behavior

**if you want to register a rust function in scheme:**
1. write an `unsafe extern "C" fn` matching chibi's calling convention:
   - 0 args: `fn(ctx, self, 0) -> sexp`
   - 1 arg: `fn(ctx, self, 1, arg1) -> sexp`
   - 2 args: `fn(ctx, self, 2, arg1, arg2) -> sexp`
   - 3 args: `fn(ctx, self, 3, arg1, arg2, arg3) -> sexp`
2. use `raw::sexp_unbox_fixnum`, `raw::sexp_flonum_value` etc. to extract args
3. use `raw::sexp_make_fixnum`, `raw::sexp_c_str` etc. to build return values
4. register with `ctx.define_fn0("name", f)` (or `define_fn1`, `define_fn2`, `define_fn3`)
5. see `examples/ffi.rs` for working examples
