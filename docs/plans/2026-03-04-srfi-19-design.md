# srfi-19 time data types — design

issue: #84

## overview

add full SRFI-19 (time data types and procedures) to tein. pure scheme
implementation adapted from the MIT-licensed reference impl, backed by
rust trampolines in `(tein time)` for system clock access.

## architecture

```
(tein time)              rust trampolines
                           existing: current-second, current-jiffy, jiffies-per-second
                           new: timezone-offset-seconds
                              ↑
(scheme time)            shadow, re-exports from (tein time)
                           (replaces current (chibi time) reimplementation)
                              ↑
(srfi 19)                pure scheme (~1500 lines, adapted reference impl)
                           imports: (scheme base), (scheme char), (scheme write),
                                    (srfi 9), (tein time)
```

## components

### 1. `(tein time)` — new trampoline

add `timezone-offset-seconds` to `time.rs`:

- returns local UTC offset in seconds (e.g. 3600 for UTC+1)
- implementation: `libc::localtime_r` → `tm_gmtoff` via `unsafe extern "C"`
- no new crate dependency — just a raw libc call
- returns real timezone even in sandbox (low-sensitivity metadata)

### 2. `(scheme time)` shadow consolidation

replace the current `(chibi time)`-based shadow with a thin re-export from
`(tein time)`:

```scheme
(define-library (scheme time)
  (import (tein time))
  (export current-second current-jiffy jiffies-per-second))
```

single source of truth. eliminates the parallel pure-scheme reimplementation
that duplicates `(tein time)` functionality.

### 3. `(srfi 19)` pure scheme implementation

adapt the MIT-licensed reference implementation (will fitzgerald, I/NET Inc).

#### porting changes

| reference impl               | tein port                                         |
|-------------------------------|---------------------------------------------------|
| `define-struct` (mzscheme)    | `define-record-type` (srfi-9)                     |
| `current-seconds` etc        | `(current-second)` from `(tein time)`, split      |
| `receive` macro (inline)     | inline (same as reference) or `(srfi 8)`          |
| `time-process` / `time-thread` | raise `'unsupported-clock-type` error           |
| `time-gc` (mzscheme ext)     | removed                                           |
| `read-leap-second-table`     | removed (file I/O, sandbox incompatible)          |
| locale strings                | kept as-is (english, hardcoded)                   |

#### leap second table

static alist baked into scheme. 27 entries from 1972–2017. the table hasn't
changed in 9 years (ITU voted to abolish leap seconds by 2035). zero
maintenance burden.

#### time types supported

- `time-utc` — full support
- `time-tai` — full support (via leap second table)
- `time-monotonic` — full support (defined as TAI, per reference impl)
- `time-duration` — full support
- `time-process` — raises `unsupported-clock-type` error
- `time-thread` — raises `unsupported-clock-type` error

#### files (in chibi fork, VFS-embedded)

- `lib/srfi/19.sld` — library definition with exports
- `lib/srfi/19.scm` — implementation

### 4. VFS registry entry

```rust
VfsEntry {
    path: "srfi/19",
    deps: &["scheme/base", "scheme/char", "scheme/write", "srfi/9", "tein/time"],
    files: &["lib/srfi/19.sld", "lib/srfi/19.scm"],
    clib: None,
    default_safe: true,
    source: VfsSource::Embedded,
    feature: Some("time"),
    shadow_sld: None,
}
```

gated behind existing `time` cargo feature (on by default, zero external deps).

### 5. tests

- scheme integration tests in `tests/scheme/srfi19.scm`
- adapt reference test suite (262 lines, MIT-licensed)
- vfs_module_tests entry if compatible with `(chibi test)` harness

## security notes

- `timezone-offset-seconds` returns real timezone in sandbox — low-sensitivity
  metadata, not a meaningful side-channel
- `time-process`/`time-thread` stubbed — no host CPU info leakage
- leap second table is static data — no file I/O
- all pure scheme math — no new unsafe code beyond the one `localtime_r` call
