# `(tein safe-regexp)` ŌĆö design document

**issue**: #37
**date**: 2026-03-04
**milestone**: 8 ŌĆö rust ecosystem bridge

## summary

linear-time regular expression support via rust's `regex` crate, exposed as `(tein safe-regexp)` with an SRFI-115-inspired API. guarantees O(n) matching ŌĆö no backtracking, no ReDoS. the right choice for sandboxed environments with untrusted patterns.

## approach

**maximal rust, minimal scheme.** `Regexp` foreign type wraps `regex::Regex` via `#[tein_type]`. core operations are `#[tein_methods]` on `Regexp` returning `Value` for match vectors. scheme `.scm` provides user-facing free-function wrappers (`regexp-search` etc.) with string-or-regexp dispatch, delegating to internal methods (`safe-regexp-search` etc.). `regexp-fold` is scheme-side (takes closures). match results are plain scheme vectors ŌĆö destructurable with `(chibi match)`.

**prerequisite**: #114 ŌĆö `gen_return_conversion` in tein-macros needs a `Value` arm for `#[tein_fn]` free fns returning `Result<Value, String>`.

## API surface

### foreign type

`Regexp` ŌĆö wraps `regex::Regex`. compiled once, reused. all API procs accept both a pre-compiled `Regexp` and a raw pattern string (compile-on-the-fly for one-shot usage).

### two-layer API

the type is named `safe-regexp`, so auto-generated method names are `safe-regexp-search`, `safe-regexp-matches`, etc. (internal). scheme wrappers provide the user-facing `regexp-search`, `regexp-matches`, etc. with string-or-regexp coercion via `(%ensure-regexp rx)`.

### rust-side ŌĆö `#[tein_methods]` on `Regexp` + `#[tein_fn]` free fns

| scheme name | rust impl | signature | notes |
|---|---|---|---|
| `regexp` | `Regex::new()` | `(regexp pattern)` ŌåÆ regexp | error string on invalid pattern |
| `regexp?` | type check | `(regexp? obj)` ŌåÆ bool | auto-generated by `#[tein_type]` |
| `regexp-matches` | `Regex::captures()` + anchoring | `(regexp-matches rx str)` ŌåÆ match-vec or `#f` | entire string must match |
| `regexp-matches?` | `Regex::is_match()` + anchoring | `(regexp-matches? rx str)` ŌåÆ bool | |
| `regexp-search` | `Regex::captures()` | `(regexp-search rx str)` ŌåÆ match-vec or `#f` | first match anywhere |
| `regexp-search-from` | `Regex::captures_at()` | `(regexp-search-from rx str start)` ŌåÆ match-vec or `#f` | offset for fold iteration |
| `regexp-replace` | `Regex::replace()` | `(regexp-replace rx str replacement)` ŌåÆ string | first match only |
| `regexp-replace-all` | `Regex::replace_all()` | `(regexp-replace-all rx str replacement)` ŌåÆ string | |

### scheme-side (`safe-regexp.scm`)

```scheme
(define (regexp-fold rx kons knil str . opts)
  ;; opts: [finish [start [end]]]
  ;; loop via regexp-search-from, kons: (i match str acc)
  ;; finish: (i match str acc), default: (lambda (i m s acc) acc)
  ...)
```

### match vector shape

```scheme
#(#("2026-03-04" 0 10)    ;; group 0 (whole match): text, start, end
  #("2026" 0 4)            ;; group 1
  #("03" 5 7)              ;; group 2
  #("04" 8 10))            ;; group 3

;; unmatched optional group => #f in that slot
```

works naturally with `(chibi match)` for destructuring:

```scheme
(import (chibi match))
(match (regexp-search "(\\d+)-(\\d+)" "foo-42-7-bar")
  (#(whole #(y _ _) #(m _ _))
   (list y m)))  ;; => ("42" "7")
```

## architecture

### file layout

- `src/safe_regexp.rs` ŌĆö `#[tein_module("safe-regexp")]` with `Regexp` type + all fns
- macro generates `lib/tein/safe-regexp.sld` + `.scm` (`.scm` contains `regexp-fold`)

### wiring

- `Cargo.toml`: feature `regex = ["dep:regex"]`, in `default`
- `lib.rs`: `#[cfg(feature = "regex")] mod safe_regexp;`
- `context.rs`: register call gated on `#[cfg(feature = "regex")]`
- `vfs_registry.rs`: `path: "tein/safe-regexp"`, `VfsSource::Dynamic`, `feature: Some("regex")`, `default_safe: true`

### string-or-regexp argument handling

scheme-side: `(%ensure-regexp rx)` compiles strings on the fly, passes `Regexp` foreign objects through. rust methods always receive a `Regexp` foreign type ŌĆö no mixed-type dispatch needed in rust.

### VFS override

macro-generated `.sld/.scm` are overridden at registration time with hand-written versions that add scheme wrappers and `regexp-fold`. native fns resolve via top-level env (eval.c patch H). docs sub-library (`tein/safe-regexp/docs`) reflects only macro-generated exports ŌĆö `regexp-fold` and wrapper names not listed there (known limitation).

### sandbox

`default_safe: true` ŌĆö pure computation, no I/O. fuel limits bound scheme-level overhead. linear-time guarantee from the regex engine itself. safe for untrusted code by design.

## byte vs char indexing

match vector `start`/`end` values are **byte offsets** (rust regex semantics). for ASCII this is invisible. for multi-byte unicode, offsets are not char-indexed. documented, not converted ŌĆö `regexp-match-submatch` gives text directly so raw offsets are rarely needed. power-user concern only.

## testing strategy

### rust unit tests

- compilation: valid, invalid, empty patterns
- match vector shape: structure, group count, offsets, `#f` for unmatched optionals
- `search` vs `matches`: partial vs full-string
- `replace` / `replace-all`: backrefs, empty match, no match
- `split`: basic, consecutive delimiters, trailing empty
- `extract`: multiple matches, groups
- `search-from`: offset handling, out-of-bounds
- string-as-pattern: every fn works with raw string, not just compiled

### scheme integration tests

- `regexp-fold`: accumulator, finish proc, start/end
- `(chibi match)` destructuring on results
- sandbox: works under `Modules::Safe`
- error cases: bad pattern, wrong types

### edge cases

- empty string, empty pattern
- unicode multi-byte (offset correctness)
- no-match ŌåÆ `#f` consistently
