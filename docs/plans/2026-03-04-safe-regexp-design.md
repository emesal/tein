## summary

linear-time regular expression support via rust's `regex` crate, exposed as `(tein safe-regexp)` with an SRFI-115-inspired API. guarantees O(n) matching â€” no backtracking, no ReDoS. the right choice for sandboxed environments with untrusted patterns.

## approach

**maximal rust, minimal scheme.** rust implements all core operations via `#[tein_module]`. scheme implements only `regexp-fold` (higher-order iteration over matches via `regexp-search-from`). match results are plain scheme vectors â€” no foreign types for matches, destructurable with `(chibi match)`.

## API surface

### foreign type

`Regexp` â€” wraps `regex::Regex`. compiled once, reused. all API procs accept both a pre-compiled `Regexp` and a raw pattern string (compile-on-the-fly for one-shot usage).

### scheme-side (`safe-regexp.scm`)

```scheme
(define (regexp-fold rx kons knil str . opts)
  ;; opts: [finish [start [end]]]
  ;; loop via regexp-search-from, kons: (i match str acc)
  ;; finish: (i match str acc), default: (lambda (i m s acc) acc)
  ...)
```

### match vector shape

```scheme
#(#("2026-03-04" 0 10)    ;; group 0 (whole match): text, start, end
  #("2026" 0 4)            ;; group 1
  #("03" 5 7)              ;; group 2
  #("04" 8 10))            ;; group 3

;; unmatched optional group => #f in that slot
```

works naturally with `(chibi match)` for destructuring:

```scheme
(import (chibi match))
(match (regexp-search "(\\d+)-(\\d+)" "foo-42-7-bar")
  (#(whole #(y _ _) #(m _ _))
   (list y m)))  ;; => ("42" "7")
```

## architecture

### file layout

- `src/safe_regexp.rs` â€” `#[tein_module("safe-regexp")]` with `Regexp` type + all fns
- macro generates `lib/tein/safe-regexp.sld` + `.scm` (`.scm` contains `regexp-fold`)

### wiring

- `Cargo.toml`: feature `regex = ["dep:regex"]`, in `default`
- `lib.rs`: `#[cfg(feature = "regex")] mod safe_regexp;`
- `context.rs`: register call gated on `#[cfg(feature = "regex")]`
- `vfs_registry.rs`: `path: "tein/safe-regexp"`, `VfsSource::Dynamic`, `feature: Some("regex")`, `default_safe: true`

### string-or-regexp argument handling

every fn accepting `rx` handles both `Value::Foreign` (extract `Regexp`) and `Value::String` (compile on the fly). shared helper, exact mechanics tbd during implementation based on `#[tein_type]` extraction patterns.

### sandbox

`default_safe: true` â€” pure computation, no I/O. fuel limits bound scheme-level overhead. linear-time guarantee from the regex engine itself. safe for untrusted code by design.

## byte vs char indexing

match vector `start`/`end` values are **byte offsets** (rust regex semantics). for ASCII this is invisible. for multi-byte unicode, offsets are not char-indexed. documented, not converted â€” `regexp-match-submatch` gives text directly so raw offsets are rarely needed. power-user concern only.

## testing strategy

### rust unit tests

- compilation: valid, invalid, empty patterns
- match vector shape: structure, group count, offsets, `#f` for unmatched optionals
- `search` vs `matches`: partial vs full-string
- `replace` / `replace-all`: backrefs, empty match, no match
- `split`: basic, consecutive delimiters, trailing empty
- `extract`: multiple matches, groups
- `search-from`: offset handling, out-of-bounds
- string-as-pattern: every fn works with raw string, not just compiled

### scheme integration tests

- `regexp-fold`: accumulator, finish proc, start/end
- `(chibi match)` destructuring on results
- sandbox: works under `Modules::Safe`
- error cases: bad pattern, wrong types

### edge cases

- empty string, empty pattern
- unicode multi-byte (offset correctness)
- no-match â†’ `#f` consistently
