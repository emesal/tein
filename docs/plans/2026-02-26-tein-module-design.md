# `#[tein_module]` proc macro design

**issue:** #40
**status:** approved design, pending implementation plan
**date:** 2026-02-26

## summary

a single `#[tein_module("name")]` attribute macro on a `mod` block that generates
everything needed to expose rust code as an idiomatic r7rs scheme module: ForeignType
impls, extern "C" wrappers, VFS `.sld`/`.scm` content, and a registration function.

replaces `#[scheme_fn]` entirely. one annotation, one source of truth.

## motivation

adding a new tein module (`(tein json)`, `(tein regex)`, etc.) currently requires:

- manual `ForeignType` impl with `type_name()` and `methods()` returning a static slice
- hand-written VFS `.sld` + `.scm` files
- manual `register_foreign_type` calls
- manual `define_fn_variadic` calls for free functions
- manual extern "C" wrappers (or `#[scheme_fn]` + manual registration)

the proc macro collapses all of this into annotated rust.

## user-facing API

```rust
#[tein_module("json")]
mod json {
    use serde_json;

    /// parse a JSON string into a json-value foreign object
    #[tein_fn]
    fn parse(s: String) -> Result<JsonValue, Error> {
        let v: serde_json::Value = serde_json::from_str(&s)
            .map_err(|e| Error::EvalError(e.to_string()))?;
        Ok(JsonValue { inner: v })
    }

    /// stringify a json-value back to a JSON string
    #[tein_fn]
    fn stringify(v: JsonValue) -> Result<String, Error> {
        Ok(v.inner.to_string())
    }

    /// the JSON value foreign type
    #[tein_type(name = "json-value")]
    struct JsonValue {
        inner: serde_json::Value,
    }

    #[tein_methods]
    impl JsonValue {
        /// get a value by key (objects) or index (arrays)
        fn get(&self, key: String) -> Result<Value, Error> { ... }

        /// check if this is a JSON object
        fn object_q(&self) -> bool { ... }
    }
}
```

registration by the caller:

```rust
let ctx = ContextBuilder::new().standard_env().build()?;
json::register_module_json(&ctx)?;
// scheme code can now: (import (tein json))
```

## naming conventions

all symbols are prefixed with the module name.

| rust | scheme | rule |
|------|--------|------|
| `fn parse` in module `"json"` | `json-parse` | module prefix + kebab-case |
| `fn is_match` in method | `regex-is-match` | `snake_case` â†’ `kebab-case` |
| `fn object_q` | `json-value-object?` | trailing `_q` â†’ `?` |
| `fn set_value_bang` | `json-value-set-value!` | trailing `_bang` â†’ `!` |
| `#[tein_fn(name = "custom")]` | `custom` | override, no auto-prefix |

type-related symbols generated automatically:
- type predicate: `regex?`, `json-value?`
- method wrappers: `regex-is-match`, `json-value-get`
- no auto-constructor â€” `#[tein_fn]` functions returning the type serve as constructors

## macro codegen

given `#[tein_module("regex")]` with a type `Regex` and fn `compile`, the macro emits:

**original items** â€” preserved unmodified inside the mod.

**`ForeignType` impl:**

```rust
impl ForeignType for Regex {
    fn type_name() -> &'static str { "regex" }
    fn methods() -> &'static [(&'static str, MethodFn)] {
        &[
            ("is-match", |obj, _ctx, args| {
                let this = obj.downcast_ref::<Regex>().unwrap();
                let text: String = /* extract from args[0] */;
                Ok(Value::Boolean(this.is_match(text)))
            }),
            // ...
        ]
    }
}
```

**extern "C" wrappers** for each `#[tein_fn]`:

```rust
unsafe extern "C" fn __tein_compile(
    ctx: tein::raw::sexp, _self: tein::raw::sexp,
    _n: tein::raw::sexp_sint_t, args: tein::raw::sexp,
) -> tein::raw::sexp {
    std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| unsafe {
        // extract args from raw sexp
        // call compile(pattern)
        // Ok(Regex) â†’ insert into ForeignStore, return Foreign sexp
        // Err â†’ return error sexp
    }))
    // ... panic boundary
}
```

when a `#[tein_fn]` returns a `T: ForeignType`, the wrapper calls
`ctx.foreign_value()` to insert into the store and return `Value::Foreign`.

**VFS content as const strings:**

```rust
pub const TEIN_MODULE_REGEX_SLD: &str = "\
    (define-library (tein regex)\n\
      (import (scheme base))\n\
      (export regex-compile regex? regex-is-match regex-find)\n\
      (include \"regex.scm\"))";

pub const TEIN_MODULE_REGEX_SCM: &str = "\
    ;; generated by #[tein_module]";
```

the `.sld` export list is assembled automatically from all `#[tein_fn]` names,
type predicates, and method wrappers.

**registration function:**

```rust
pub fn register_module_regex(ctx: &Context) -> tein::Result<()> {
    ctx.register_vfs_module("lib/tein/regex.sld", TEIN_MODULE_REGEX_SLD)?;
    ctx.register_vfs_module("lib/tein/regex.scm", TEIN_MODULE_REGEX_SCM)?;
    ctx.register_foreign_type::<Regex>()?;
    ctx.define_fn_variadic("regex-compile", __tein_compile)?;
    Ok(())
}
```

## runtime VFS registration

new API on Context:

```rust
/// register a virtual filesystem entry at runtime.
///
/// path is relative to the VFS root, e.g. "lib/tein/json.sld".
/// content is the file content as a string. the entry becomes
/// available to chibi's module resolver immediately.
pub fn register_vfs_module(&self, path: &str, content: &str) -> Result<()>
```

C-side addition in `tein_shim.c`:

```c
void tein_vfs_register(sexp ctx, const char* path, const char* content, int len);
```

hooks into the same VFS lookup mechanism as compile-time entries. existing
hand-written VFS modules (`(tein foreign)`, `(tein reader)`, etc.) are unaffected.

ordering constraint: `register_module_*(ctx)` must be called before any scheme
code does `(import (tein ...))`.

## type mapping

### `#[tein_fn]` args â€” extracted from raw sexp (extern "C" boundary)

| rust type | scheme type | extraction |
|-----------|-------------|------------|
| `i64` | integer | `sexp_unbox_fixnum` |
| `f64` | float | `sexp_flonum_value` (with int coercion) |
| `String` | string | `sexp_string_data` â†’ UTF-8 |
| `bool` | boolean | `== get_true()` |
| `Value` | any | `Value::from_raw()` passthrough |
| `T: ForeignType` | foreign object | extract handle â†’ store lookup â†’ downcast |

<!-- extensibility: to add new types (e.g. Vec<Value>, &[u8], char),
     add a branch in gen_arg_extraction() in lib.rs matching the type name
     and emitting the appropriate extraction code. follow the existing
     patterns â€” each branch produces a proc_macro2::TokenStream that
     declares a local variable with the extracted value. -->

### `#[tein_methods]` args â€” extracted from `&[Value]` (inside dispatch chain)

| rust type | from `Value` |
|-----------|-------------|
| `i64` | `value.as_integer()` |
| `f64` | `value.as_float()` |
| `String` | `value.as_string()` |
| `bool` | `value.as_boolean()` |
| `Value` | clone/move directly |
| `T: ForeignType` | `value.as_foreign()` â†’ store lookup â†’ downcast |

<!-- extensibility: to add new types here, add a branch in
     gen_method_arg_extraction() matching the type name and emitting
     the appropriate Value::as_*() call. same pattern as above. -->

### return types â€” reverse mapping, plus:

- `Result<T, Error>` â†’ unwrap or propagate as scheme error
- `()` â†’ `Value::Unspecified`
- `T: ForeignType` â†’ insert into store, return `Value::Foreign`

<!-- extensibility: to add new return types, add a branch in
     gen_return_conversion(). for types returning containers
     (e.g. Vec<Value> â†’ scheme list), emit the appropriate
     Value constructor call. -->

### `&self` vs `&mut self`

`#[tein_methods]` supports both. `&self` â†’ `downcast_ref`, `&mut self` â†’
`downcast_mut`. matches the existing `MethodFn` signature.

## `#[tein_fn]` standalone mode

`#[tein_fn]` works in two modes:

1. **inside `#[tein_module]`** â€” module macro handles everything
2. **standalone** â€” generates the extern "C" wrapper, caller manually calls
   `define_fn_variadic`. identical to current `#[scheme_fn]` behaviour.

this lets us remove `#[scheme_fn]` entirely. migration is mechanical:
`#[scheme_fn]` â†’ `#[tein_fn]`, wrapper prefix `__tein_` stays the same.

## error handling

**compile-time** (proc macro errors with clear spans):

- `#[tein_module]` on a non-`mod` item
- `#[tein_fn]` with unsupported arg type (message lists supported types)
- `#[tein_methods]` on impl for a type without `#[tein_type]`
- duplicate type names within a module

**runtime** (scheme-side):

- wrong arg type: `"json-parse: expected string, got integer"`
- wrong arg count: `"json-parse: expected 1 argument, got 3"`
- method on wrong type: existing foreign dispatch error messages (LLM-friendly)

## testing strategy

- **trybuild tests** in tein-macros â€” compile-time error cases
- **integration test module** â€” `#[tein_module("test-mod")]` exercising all
  features: free fns, types, methods, `&self`/`&mut self`, all type mappings,
  name overrides, `_q`/`_bang` transforms
- **scheme-side test** â€” `.scm` file testing the generated module via
  `(import (tein test-mod))`
- **migration test** â€” all 12 existing `#[scheme_fn]` tests pass with `#[tein_fn]`

## decisions and alternatives

**approach A (chosen): single `#[tein_module]` on mod block**
- one annotation, single source of truth
- mod block is the natural grouping unit
- no "forgot to register" failure mode

**approach B (rejected): independent attributes + `register_module!` macro**
- module name repeated on every annotation
- manual listing in registration macro â€” can forget items
- complexity spread across multiple macros instead of one

**VFS: runtime registration (chosen) over build.rs coordination**
- proc macros run after build.rs â€” can't write files for build.rs to pick up
- runtime `register_vfs_module` is a small, composable addition
- existing compile-time VFS modules unaffected
