# (tein introspect) — environment introspection API

**issues**: #27, #83
**date**: 2026-03-12
**status**: design

## summary

add `(tein introspect)`, a scheme module providing runtime environment introspection. lets LLM agents discover what modules are available, what each module exports, what bindings are in scope, and what shape procedures have — all from within the scheme environment. also provides a structured data dump suitable for prompt injection.

closes two issues: #27 (VFS-embedded documentation for LLM schemers) and #83 (environment introspection API for LLM agents).

## motivation

an LLM writing scheme code inside a sandboxed tein context currently has no way to ask "what's available to me?" from within the environment itself. it can't list importable modules, inspect exports, or query procedure arity. the existing `(tein docs)` module handles doc alist formatting but has no awareness of the module system or live environment.

the LLM's harness communicates via `scheme_eval` — so the solution is to make all introspection accessible as scheme procedure calls. the harness can query `(describe-environment)` for a full overview to inject into the LLM's prompt, and the LLM can use individual introspection functions as it writes code.

## design

### module registration

`(tein introspect)` is registered as:

```rust
VfsEntry {
    path: "tein/introspect",
    // scheme/write needed by describe-environment/text for display/write
    deps: &["scheme/base", "scheme/write", "tein/docs"],
    files: &["lib/tein/introspect.sld", "lib/tein/introspect.scm"],
    clib: None,
    default_safe: true,
    source: VfsSource::Embedded,
    feature: None,
    shadow_sld: None,
}
```

`default_safe: true` — the primary consumer is LLMs in sandboxed contexts. the module exposes only read-only introspection; it grants no new capabilities beyond seeing what the sandbox already provides.

a hand-written docs sub-library is also registered:

```rust
VfsEntry {
    path: "tein/introspect/docs",
    deps: &["scheme/base"],
    files: &["lib/tein/introspect/docs.sld", "lib/tein/introspect/docs.scm"],
    clib: None,
    default_safe: true,
    source: VfsSource::Embedded,
    feature: None,
    shadow_sld: None,
}
```

the docs sub-library exports `introspect-docs`, matching the convention established by `#[tein_module]`-generated doc sub-libraries.

### implementation approach: layered hybrid

each function lives where it naturally fits:

- **rust trampolines** for module-level data (VFS_ALLOWLIST, MODULE_EXPORTS — already available as thread-locals and build-generated statics)
- **C shims** for env-level data (env chain walking, lambda formals, opcode arity — requires chibi internals)
- **scheme** for composition and formatting (binding-info, describe-environment — glue logic on top of the primitives)

### API

#### `(available-modules)` — rust trampoline

```scheme
(available-modules)
;; → ((scheme base) (scheme write) (tein json) (tein introspect) ...)
```

returns a list of module paths importable in the current context, as proper lists matching `(import ...)` syntax.

- **sandboxed contexts**: returns modules in `VFS_ALLOWLIST` (converted from `"scheme/base"` → `(scheme base)`)
- **unsandboxed contexts**: returns all `VFS_REGISTRY` paths plus any dynamically registered modules

#### `(imported-modules)` — C shim + rust wrapper

```scheme
(imported-modules)
;; → ((scheme base) (scheme write) (tein introspect))
```

returns modules that have actually been imported in the current context. walks chibi's `*modules*` alist in the meta env, filtering for entries where `module-env` is non-#f (loaded). in sandboxed contexts, results are further filtered to the `VFS_ALLOWLIST` to prevent information leakage about internal module state.

#### `(module-exports '(module name))` — rust trampoline

```scheme
(module-exports '(scheme base))    ;; → (map for-each values apply ...)
(module-exports '(tein json))      ;; → (json-parse json-stringify)
```

returns a list of exported binding name symbols for the given module. primary source is the `MODULE_EXPORTS` table generated by `build.rs` at compile time. for dynamically registered modules (via `ctx.register_module()`) not in `MODULE_EXPORTS`, falls back to chibi's own module export mechanism via the meta env — looks up the module in `*modules*` and reads its export list.

raises an error if the module is not in the current allowlist (no peeking at modules the sandbox doesn't grant access to). raises an error if the module is not found in either `MODULE_EXPORTS` or chibi's module registry.

#### `(env-bindings)` / `(env-bindings prefix)` — C shim

```scheme
(env-bindings)              ;; → ((map . procedure) (+ . procedure) (x . variable) ...)
(env-bindings "my-")        ;; → ((my-var . variable) (my-fn . procedure))
```

returns an alist of `(name . kind)` pairs for all bindings visible in the current environment. walks `sexp_env_bindings` + `sexp_env_parent` chain up to (but not including) the global env boundary. kind is one of: `procedure`, `syntax`, `variable`.

**scope**: walks the full parent chain starting from `sexp_context_env(ctx)`. in sandboxed contexts this is the null env → only explicitly granted bindings + user defines are visible. in unsandboxed standard-env contexts this includes stdlib bindings from parent envs. shadowed bindings are deduplicated — the innermost (most local) binding wins.

optional prefix argument filters by symbol name prefix (done in C for performance — avoids scheme-level string ops in a loop over potentially hundreds of bindings). the prefix match compares against the symbol's string representation obtained via `sexp_symbol_to_string`.

#### `(procedure-arity proc)` — C shim

```scheme
(procedure-arity map)       ;; → (1 . #f)    ; min 1, variadic
(procedure-arity cons)      ;; → (2 . 2)     ; exactly 2
(procedure-arity +)         ;; → (0 . #f)    ; variadic from 0
```

returns `(min . max)` where max is `#f` if the procedure is variadic.

- **compiled lambdas** (`sexp_procedurep`): reads `sexp_procedure_num_args` + `sexp_procedure_variadic_p`
- **opcodes** (`sexp_opcodep`): reads `sexp_opcode_num_args` + `sexp_opcode_variadic_p`
- **non-procedures**: returns `#f`

**known limitation**: procedures registered via `define_fn_variadic` (all tein trampolines) report `(0 . #f)` — the actual arity is enforced in rust, not in chibi's metadata. `binding-info` compensates by preferring doc alist metadata over raw arity for tein modules when available.

#### `(binding-info sym)` — scheme composition

```scheme
(binding-info 'map)
;; → ((name . map)
;;    (kind . procedure)
;;    (arity 1 . #f)
;;    (module . (scheme base)))

(binding-info 'json-parse)
;; → ((name . json-parse)
;;    (kind . procedure)
;;    (arity 1 . #f)
;;    (module . (tein json))
;;    (doc . "parse a json string into scheme values"))
```

composes the lower-level primitives: `tein_binding_kind` for kind, `tein_procedure_arity` for arity. the `module` field uses a reverse index (binding name → providing module) built once at import time by inverting `MODULE_EXPORTS`. for tein modules with doc sub-libraries, includes the `doc` field by looking up the binding in the appropriate `X-docs` alist.

returns `#f` if the symbol is not bound in the current environment.

#### `(describe-environment)` — scheme composition

```scheme
(describe-environment)
;; → ((modules
;;      ((name . (scheme base))
;;       (exports . (map for-each values ...)))
;;      ((name . (tein json))
;;       (exports . (json-parse json-stringify))
;;       (docs . ((json-parse . "parse a json string ...")
;;                (json-stringify . "convert scheme values ..."))))
;;      ...))
```

iterates `(available-modules)`, calls `(module-exports mod)` for each. for tein modules, includes doc alists loaded eagerly at `(tein introspect)` import time (see "doc alist loading" section below). returns structured scheme data.

note: doc sub-libraries for tein modules are already accessible in sandboxed contexts because the VFS gate's prefix matching allows `tein/X/docs` when `tein/X` is in the allowlist (the check is `suffix.starts_with(prefix)`).

modules only — does not include user-defined bindings. call `(env-bindings)` separately for that.

#### `(describe-environment/text)` — scheme composition

```scheme
(describe-environment/text)
;; → "(tein introspect) — environment overview\n\n14 modules available:\n\n(scheme base) — 142 exports\n  map, for-each, values, ...\n\n(tein json) — 2 exports\n  json-parse — parse a json string ...\n  json-stringify — convert scheme values ...\n\n..."
```

pretty-prints the output of `describe-environment` into a human/LLM-readable string.

#### `introspect-docs` — hand-written constant

```scheme
introspect-docs
;; → ((__module__ . "tein introspect")
;;    (available-modules . "list modules importable in current context")
;;    (imported-modules . "list modules already imported in current context")
;;    (module-exports . "list exported binding names of a module")
;;    (env-bindings . "list all bindings in current environment, optional prefix filter")
;;    (binding-info . "detailed info about a binding: kind, arity, module, docs")
;;    (procedure-arity . "return (min . max) arity, #f for max if variadic")
;;    (describe-environment . "structured data dump of all available modules and exports")
;;    (describe-environment/text . "pretty-printed text overview of the environment"))
```

### C shims in `tein_shim.c`

four new functions in the chibi fork's `tein_shim.c`. all shims must use `sexp_gc_var`/`sexp_gc_preserve`/`sexp_gc_release` for GC rooting — chibi's conservative stack scanning is disabled in tein, so any `sexp` held across an allocating call (`sexp_cons`, `sexp_intern`, `sexp_symbol_to_string`) can be freed.

#### `tein_env_bindings_list(ctx, prefix)`

walks `sexp_env_bindings(sexp_context_env(ctx))` and parent chain. each binding cell is `(name . value)`. determines kind by inspecting the value: `sexp_procedurep` or `sexp_opcodep` → `procedure`, `sexp_syntacticp` → `syntax`, else → `variable`. if `prefix` is non-NULL, filters by prefix match on the symbol's string representation (obtained via `sexp_symbol_to_string(ctx, sym)`, then compared with `sexp_string_data`). returns a scheme list of `(name-symbol . kind-symbol)` pairs.

**GC rooting**: the accumulator list and kind symbols (`procedure`, `syntax`, `variable`) must be GC-rooted across the loop. intern the three kind symbols once before the loop and root them.

**deduplication**: track seen symbol names. if a binding name was already seen (from an inner env), skip it — innermost wins.

#### `tein_procedure_arity(ctx, proc)`

for `sexp_procedurep`: returns `sexp_cons(sexp_make_fixnum(sexp_procedure_num_args(proc)), variadic ? SEXP_FALSE : sexp_make_fixnum(sexp_procedure_num_args(proc)))`. for `sexp_opcodep`: same logic with `sexp_opcode_num_args` / `sexp_opcode_variadic_p`. for anything else: returns `SEXP_FALSE`.

**GC rooting**: minimal — `sexp_make_fixnum` does not allocate (fixnums are immediate). `sexp_cons` allocates once; root the fixnum args if needed.

#### `tein_imported_modules_list(ctx)`

accesses `sexp_global(ctx, SEXP_G_META_ENV)` to get the meta env, then looks up the `*modules*` binding via `sexp_intern(ctx, "*modules*")` + `sexp_env_ref`. walks the resulting alist, collecting module names (car of each entry) where `module-env` (`sexp_vector_ref(mod, 1)`) is non-`SEXP_FALSE`. returns a scheme list of module name lists.

**GC rooting**: the `*modules*` symbol, the meta-env lookup result, and the accumulator list must be rooted. `sexp_intern` is an allocating call.

**sandbox filtering**: the returned list is post-filtered in the rust wrapper (not in C) against `VFS_ALLOWLIST` for sandboxed contexts.

#### `tein_binding_kind(ctx, sym)`

looks up `sym` in `sexp_context_env(ctx)` via `sexp_env_ref`. classifies the value and returns a kind symbol (`procedure`, `syntax`, `variable`, `unknown`).

**GC rooting**: minimal — `sexp_env_ref` doesn't allocate, `sexp_intern` for the kind symbol does.

### trampoline registration

all trampolines registered into the **primitive env** during `build()`, before `load_standard_env`. this follows the established pattern from `(tein filesystem)` and `(tein process)` — ensures trampolines end up in `*chibi-env*` and are visible to the library body via `(import (chibi))`.

internal names follow the convention: `tein-available-modules-internal`, `tein-module-exports-internal`, etc. the `.sld` exports them under clean public names via `(export (rename ...))`.

### doc alist loading

doc alists from tein module sub-libraries are loaded **eagerly at `(tein introspect)` import time**, not dynamically via `eval`. the `.scm` body imports each `(tein X docs)` sub-library for tein modules in the allowlist, using `guard` to silently skip on failure. this avoids a runtime dependency on `(scheme eval)` and works in all sandbox configurations.

the loaded doc alists are stored in a module-level alist mapping module path → doc alist. `describe-environment` and `binding-info` read from this cache.

**note on import-time cost**: for `Modules::All` contexts, this eagerly imports all tein doc sub-libraries. the cost is O(number of tein modules) — each sub-library is a single `define` of a quoted alist. this is paid once at `(import (tein introspect))` time and is acceptable for the LLM agent use case.

### reverse index for `binding-info` module provenance

at `(tein introspect)` import time, the `.scm` builds a reverse index: symbol → module path. iterates `(available-modules)`, calls `(module-exports mod)` for each, and populates an alist mapping each export name to its providing module. stored in a module-level `define`.

**cost**: O(total exports across all available modules). for `Modules::All` this could be ~1000+ entries across 100+ modules, building a large alist. this is paid once at import time. if performance becomes a concern, the index could be deferred to first use via a thunk.

for names appearing in multiple modules (e.g. re-exports), the first match wins — which is the module that appears first in `(available-modules)`. this is best-effort; the `module` field in `binding-info` is informational, not authoritative.

### sandbox invariants

- `available-modules` respects the VFS allowlist — sandboxed contexts only see their granted modules
- `module-exports` rejects lookups for modules outside the allowlist
- `imported-modules` filters results to the allowlist in sandboxed contexts
- `env-bindings` walks the sandbox's null env — only sees explicitly granted bindings
- `describe-environment` composes the above, inheriting all restrictions
- no new capabilities are granted — the module is read-only introspection

### interaction with `(tein docs)`

`(tein introspect)` depends on `(tein docs)` for the `module-doc` and `module-docs` accessors. `describe-environment` and `binding-info` use these to extract docstrings from tein module doc alists.

the doc sub-library convention (`(tein X docs)` exports `X-docs`) is leveraged but not modified. `describe-environment` uses eagerly-loaded doc alists for tein modules. for chibi/srfi modules (no doc alists), output is export-names-only — the LLM already knows r7rs.

### known limitations

- **`define_fn_variadic` arity**: all tein trampolines report `(0 . #f)` from `procedure-arity` since actual arity is enforced in rust, not chibi metadata. `binding-info` compensates by preferring doc metadata when available.
- **`imported-modules` relies on chibi's `*modules*`**: accessed via `sexp_intern` + `sexp_env_ref` on the meta env. this is a chibi internal naming convention — fragile across chibi version changes, but tein pins its fork so this is acceptable.
- **`module-exports` for dynamic modules**: primary source is compile-time `MODULE_EXPORTS`. dynamically registered modules fall back to chibi's module registry — which may not have the module loaded yet if it hasn't been imported. in that case, returns an error.
- **reverse index is best-effort**: binding → module mapping doesn't handle re-exports or aliased imports. the `module` field in `binding-info` is informational.

## testing strategy

### unit tests (`context.rs`)

- `available-modules` returns correct list for `Modules::Safe` vs `Modules::All` vs `Modules::None` vs unsandboxed
- `module-exports` returns correct symbols for known modules
- `module-exports` raises error for modules outside allowlist
- `module-exports` raises error for nonexistent modules
- `procedure-arity` on compiled lambdas (fixed, variadic)
- `procedure-arity` on builtins/opcodes
- `procedure-arity` on non-procedures returns `#f`
- `env-bindings` returns expected bindings for standard env
- `env-bindings` prefix filtering works
- `imported-modules` reflects actually-imported modules
- `imported-modules` in sandbox only shows allowed modules

### integration tests (`tests/scheme/*.scm`)

- full API exercise from scheme: import, call each function, verify shapes
- `binding-info` returns correct kind/arity/module/doc fields
- `describe-environment` returns well-formed nested alist
- `describe-environment/text` returns non-empty string
- doc alist integration: tein modules include docs, chibi modules don't

### edge cases

- `env-bindings` after user `(define ...)` includes the new binding
- `binding-info` on syntax (e.g. `if`, `define`) returns `kind: syntax`
- `binding-info` on an undefined symbol
- `module-exports` on a dynamically registered module (if `(tein modules)` is allowed)
- `describe-environment` in `Modules::None` context (empty module list)
