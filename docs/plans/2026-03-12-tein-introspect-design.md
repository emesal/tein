# (tein introspect) ‚Äî environment introspection API

**issues**: #27, #83
**date**: 2026-03-12
**status**: design

## summary

add `(tein introspect)`, a scheme module providing runtime environment introspection. lets LLM agents discover what modules are available, what each module exports, what bindings are in scope, and what shape procedures have ‚Äî all from within the scheme environment. also provides a structured data dump suitable for prompt injection.

closes two issues: #27 (VFS-embedded documentation for LLM schemers) and #83 (environment introspection API for LLM agents).

## motivation

an LLM writing scheme code inside a sandboxed tein context currently has no way to ask "what's available to me?" from within the environment itself. it can't list importable modules, inspect exports, or query procedure arity. the existing `(tein docs)` module handles doc alist formatting but has no awareness of the module system or live environment.

the LLM's harness communicates via `scheme_eval` ‚Äî so the solution is to make all introspection accessible as scheme procedure calls. the harness can query `(describe-environment)` for a full overview to inject into the LLM's prompt, and the LLM can use individual introspection functions as it writes code.

## design

### module registration

`(tein introspect)` is registered as:

```rust
VfsEntry {
    path: "tein/introspect",
    deps: &["scheme/base", "scheme/write", "tein/docs"],
    files: &["lib/tein/introspect.sld", "lib/tein/introspect.scm"],
    clib: None,
    default_safe: true,
    source: VfsSource::Embedded,
    feature: None,
    shadow_sld: None,
}
```

`default_safe: true` ‚Äî the primary consumer is LLMs in sandboxed contexts. the module exposes only read-only introspection; it grants no new capabilities beyond seeing what the sandbox already provides.

a hand-written docs sub-library is also registered:

```rust
VfsEntry {
    path: "tein/introspect/docs",
    deps: &["scheme/base"],
    files: &["lib/tein/introspect/docs.sld", "lib/tein/introspect/docs.scm"],
    clib: None,
    default_safe: true,
    source: VfsSource::Embedded,
    feature: None,
    shadow_sld: None,
}
```

the docs sub-library exports `introspect-docs`, matching the convention established by `#[tein_module]`-generated doc sub-libraries.

### implementation approach: layered hybrid

each function lives where it naturally fits:

- **rust trampolines** for module-level data (VFS_ALLOWLIST, MODULE_EXPORTS ‚Äî already available as thread-locals and build-generated statics)
- **C shims** for env-level data (env chain walking, lambda formals, opcode arity ‚Äî requires chibi internals)
- **scheme** for composition and formatting (binding-info, describe-environment ‚Äî glue logic on top of the primitives)

### API

#### `(available-modules)` ‚Äî rust trampoline

```scheme
(available-modules)
;; ‚Üí ((scheme base) (scheme write) (tein json) (tein introspect) ...)
```

returns a list of module paths importable in the current context, as proper lists matching `(import ...)` syntax.

- **sandboxed contexts**: returns modules in `VFS_ALLOWLIST` (converted from `"scheme/base"` ‚Üí `(scheme base)`)
- **unsandboxed contexts**: returns all `VFS_REGISTRY` paths plus any dynamically registered modules

#### `(imported-modules)` ‚Äî C shim + rust wrapper

```scheme
(imported-modules)
;; ‚Üí ((scheme base) (scheme write) (tein introspect))
```

returns modules that have actually been imported in the current context. walks chibi's `*modules*` alist in the meta env, filtering for entries where `module-env` is non-#f (loaded). in sandboxed contexts, results are further filtered to the `VFS_ALLOWLIST` to prevent information leakage about internal module state.

#### `(module-exports '(module name))` ‚Äî rust trampoline

```scheme
(module-exports '(scheme base))    ;; ‚Üí (map for-each values apply ...)
(module-exports '(tein json))      ;; ‚Üí (json-parse json-stringify)
```

returns a list of exported binding name symbols for the given module. uses the `MODULE_EXPORTS` table generated by `build.rs` at compile time.

raises an error if the module is not in the current allowlist (no peeking at modules the sandbox doesn't grant access to). raises an error if the module is not found.

#### `(env-bindings)` / `(env-bindings prefix)` ‚Äî C shim

```scheme
(env-bindings)              ;; ‚Üí ((map . procedure) (+ . procedure) (x . variable) ...)
(env-bindings "my-")        ;; ‚Üí ((my-var . variable) (my-fn . procedure))
```

returns an alist of `(name . kind)` pairs for all bindings in the current environment. walks `sexp_env_bindings` + `sexp_env_parent` chain. kind is one of: `procedure`, `syntax`, `variable`.

optional prefix argument filters by symbol name prefix (done in C for performance ‚Äî avoids scheme-level string ops in a loop over potentially hundreds of bindings).

#### `(procedure-arity proc)` ‚Äî C shim

```scheme
(procedure-arity map)       ;; ‚Üí (1 . #f)    ; min 1, variadic
(procedure-arity cons)      ;; ‚Üí (2 . 2)     ; exactly 2
(procedure-arity +)         ;; ‚Üí (0 . #f)    ; variadic from 0
```

returns `(min . max)` where max is `#f` if the procedure is variadic.

- **compiled lambdas** (`sexp_procedurep`): reads `sexp_procedure_num_args` + `sexp_procedure_variadic_p`
- **opcodes** (`sexp_opcodep`): reads `sexp_opcode_num_args` + `sexp_opcode_variadic_p`
- **non-procedures**: returns `#f`

#### `(binding-info sym)` ‚Äî scheme composition

```scheme
(binding-info 'map)
;; ‚Üí ((name . map)
;;    (kind . procedure)
;;    (arity 1 . #f)
;;    (module . (scheme base)))

(binding-info 'json-parse)
;; ‚Üí ((name . json-parse)
;;    (kind . procedure)
;;    (arity 1 . #f)
;;    (module . (tein json))
;;    (doc . "parse a json string into scheme values"))
```

composes the lower-level primitives: `tein_binding_kind` for kind, `tein_procedure_arity` for arity. the `module` field uses a reverse index (binding name ‚Üí providing module) built once at import time by inverting `MODULE_EXPORTS`. for tein modules with doc sub-libraries, includes the `doc` field by looking up the binding in the appropriate `X-docs` alist.

#### `(describe-environment)` ‚Äî scheme composition

```scheme
(describe-environment)
;; ‚Üí ((modules
;;      ((name . (scheme base))
;;       (exports . (map for-each values ...)))
;;      ((name . (tein json))
;;       (exports . (json-parse json-stringify))
;;       (docs . ((json-parse . "parse a json string ...")
;;                (json-stringify . "convert scheme values ..."))))
;;      ...))
```

iterates `(available-modules)`, calls `(module-exports mod)` for each. for tein modules, attempts to load the doc alist from `(tein X docs)` via the established convention. returns structured scheme data.

note: doc sub-libraries for tein modules are already accessible in sandboxed contexts because the VFS gate's prefix matching allows `tein/X/docs` when `tein/X` is in the allowlist (the check is `suffix.starts_with(prefix)`).

modules only ‚Äî does not include user-defined bindings. call `(env-bindings)` separately for that.

#### `(describe-environment/text)` ‚Äî scheme composition

```scheme
(describe-environment/text)
;; ‚Üí "(tein introspect) ‚Äî environment overview\n\n14 modules available:\n\n(scheme base) ‚Äî 142 exports\n  map, for-each, values, ...\n\n(tein json) ‚Äî 2 exports\n  json-parse ‚Äî parse a json string ...\n  json-stringify ‚Äî convert scheme values ...\n\n..."
```

pretty-prints the output of `describe-environment` into a human/LLM-readable string.

#### `introspect-docs` ‚Äî hand-written constant

```scheme
introspect-docs
;; ‚Üí ((__module__ . "tein introspect")
;;    (available-modules . "list modules importable in current context")
;;    (imported-modules . "list modules already imported in current context")
;;    (module-exports . "list exported binding names of a module")
;;    (env-bindings . "list all bindings in current environment, optional prefix filter")
;;    (binding-info . "detailed info about a binding: kind, arity, module, docs")
;;    (procedure-arity . "return (min . max) arity, #f for max if variadic")
;;    (describe-environment . "structured data dump of all available modules and exports")
;;    (describe-environment/text . "pretty-printed text overview of the environment"))
```

### C shims in `tein_shim.c`

four new functions in the chibi fork's `tein_shim.c`:

#### `tein_env_bindings_list(ctx, prefix)`

walks `sexp_env_bindings(sexp_context_env(ctx))` and parent chain. each binding cell is `(name . value)`. determines kind by inspecting the value: `sexp_procedurep` or `sexp_opcodep` ‚Üí `procedure`, `sexp_syntacticp` ‚Üí `syntax`, else ‚Üí `variable`. if `prefix` is non-NULL, filters by `sexp_string_data` prefix match on the symbol name. returns a scheme list of `(name-symbol . kind-symbol)` pairs.

#### `tein_procedure_arity(ctx, proc)`

for `sexp_procedurep`: returns `sexp_cons(sexp_make_fixnum(sexp_procedure_num_args(proc)), variadic ? SEXP_FALSE : sexp_make_fixnum(sexp_procedure_num_args(proc)))`. for `sexp_opcodep`: same logic with `sexp_opcode_num_args` / `sexp_opcode_variadic_p`. for anything else: returns `SEXP_FALSE`.

#### `tein_imported_modules_list(ctx)`

accesses `sexp_global(ctx, SEXP_G_META_ENV)`, looks up `*modules*` binding. walks the alist, collecting module names (car of each entry) where `module-env` (vector-ref 1) is non-#f. returns a scheme list of module name lists.

#### `tein_binding_kind(ctx, sym)`

looks up `sym` in `sexp_context_env(ctx)` via `sexp_env_ref`. classifies the value and returns a kind symbol.

### trampoline registration

all trampolines registered into the **primitive env** during `build()`, before `load_standard_env`. this follows the established pattern from `(tein filesystem)` and `(tein process)` ‚Äî ensures trampolines end up in `*chibi-env*` and are visible to the library body via `(import (chibi))`.

internal names follow the convention: `tein-available-modules-internal`, `tein-module-exports-internal`, etc. the `.sld` exports them under clean public names via `(export (rename ...))`.

### reverse index for `binding-info` module provenance

at `(tein introspect)` import time, the `.scm` builds a reverse index: symbol ‚Üí module path. iterates `(available-modules)`, calls `(module-exports mod)` for each, and populates an alist mapping each export name to its providing module. stored in a module-level `define`.

for names appearing in multiple modules (e.g. re-exports), the first match wins ‚Äî which is the module that appears first in `(available-modules)`. this is best-effort; the `module` field in `binding-info` is informational, not authoritative.

### sandbox invariants

- `available-modules` respects the VFS allowlist ‚Äî sandboxed contexts only see their granted modules
- `module-exports` rejects lookups for modules outside the allowlist
- `imported-modules` filters results to the allowlist in sandboxed contexts
- `env-bindings` walks the sandbox's null env ‚Äî only sees explicitly granted bindings
- `describe-environment` composes the above, inheriting all restrictions
- no new capabilities are granted ‚Äî the module is read-only introspection

### interaction with `(tein docs)`

`(tein introspect)` depends on `(tein docs)` for the `module-doc` and `module-docs` accessors. `describe-environment` and `binding-info` use these to extract docstrings from tein module doc alists.

the doc sub-library convention (`(tein X docs)` exports `X-docs`) is leveraged but not modified. `describe-environment` dynamically imports doc sub-libraries for tein modules in the allowlist. for chibi/srfi modules (no doc alists), output is export-names-only ‚Äî the LLM already knows r7rs.

## testing strategy

### unit tests (`context.rs`)

- `available-modules` returns correct list for `Modules::Safe` vs `Modules::All` vs `Modules::None` vs unsandboxed
- `module-exports` returns correct symbols for known modules
- `module-exports` raises error for modules outside allowlist
- `module-exports` raises error for nonexistent modules
- `procedure-arity` on compiled lambdas (fixed, variadic)
- `procedure-arity` on builtins/opcodes
- `procedure-arity` on non-procedures returns `#f`
- `env-bindings` returns expected bindings for standard env
- `env-bindings` prefix filtering works
- `imported-modules` reflects actually-imported modules
- `imported-modules` in sandbox only shows allowed modules

### integration tests (`tests/scheme/*.scm`)

- full API exercise from scheme: import, call each function, verify shapes
- `binding-info` returns correct kind/arity/module/doc fields
- `describe-environment` returns well-formed nested alist
- `describe-environment/text` returns non-empty string
- doc alist integration: tein modules include docs, chibi modules don't

### edge cases

- `env-bindings` after user `(define ...)` includes the new binding
- `binding-info` on syntax (e.g. `if`, `define`) returns `kind: syntax`
- `binding-info` on an undefined symbol
- `module-exports` on a dynamically registered module (if `(tein modules)` is allowed)
- `describe-environment` in `Modules::None` context (empty module list)
